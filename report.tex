\noindent\textit{One-paragraph pitch.} CourtEdge is an analytics platform for \textbf{serious
recreational NBA bettors} in the US legal-sports-betting market. It does \textbf{not} sell
``guaranteed picks.'' It ingests NBA stats and multi-sportsbook odds, runs a \emph{calibrated}
win-probability model, and helps the user bet smarter through four defensible features:
\textbf{edge detection} (model probability vs.\ market-implied probability), \textbf{line
shopping} (best price across books), \textbf{CLV tracking} (Closing Line Value --- the real
long-term skill metric), and \textbf{Kelly bankroll sizing}. The monetized asset is the
\emph{refined data product}: odds + model probabilities + edge, surfaced in one screen.

\begin{mdframed}
\small \textbf{Why this framing is defensible.} The NBA market is highly efficient, so a claim
of ``we beat the market'' is hard to defend. CourtEdge instead promises to help users
\emph{get the best price, track their CLV, and size bets correctly} --- which provably
saves/makes money and sidesteps the efficiency trap --- while the calibrated model stays the
technical centerpiece. We make \textbf{no profit guarantee}; we report calibration and CLV
honestly.
\end{mdframed}

\section{Target Customer}

\paragraph{Primary segment.} US-based \emph{serious recreational} NBA bettors: people who
already bet regularly (accounts at two or more sportsbooks), bet for profit rather than pure
entertainment, but are not professional syndicates. After the US Supreme Court struck down
PASPA in May 2018, legal sports betting expanded to roughly 38--39 states plus DC, creating a
large, growing, English-speaking market essentially from scratch.

\paragraph{The job they are trying to do today.}
\begin{itemize}[nosep]
  \item Manually check odds across DraftKings / FanDuel / BetMGM / etc.\ to find the best line.
  \item Guess whether a line is ``good value'' without a probabilistic baseline.
  \item Track bets in a spreadsheet --- most do not track CLV at all.
  \item Size bets by gut feel rather than bankroll math.
\end{itemize}

\paragraph{Current workarounds.} Spreadsheets, free Reddit picks, scattered odds-comparison
sites, and expensive incumbents (OddsJam, Action Network, Unabated) at roughly \$10--150/mo.

\paragraph{Why CourtEdge is better than the status quo.} One place that unifies model edge +
line shopping + CLV tracking + Kelly sizing, with a \emph{transparent, auditable} methodology
(no ``trust me'' black box). It is cheaper and more focused than the premium incumbents and
positions on honesty rather than hype.

\paragraph{Secondary segment (future).} Sell the model-probability + edge \textbf{data feed via
API} to other tool builders (B2B2X). Out of scope for v1 but noted for go-to-market.

\section{Go-to-Market Difficulties}
\label{sec:gtm}

\begin{itemize}[nosep]
  \item \textbf{Market efficiency / trust.} NBA closing lines are near-optimal, so we must
    \emph{not} promise profit. Trust is earned via a transparent, auditable CLV track record
    and open methodology --- the differentiator from the ``tout'' scam industry.
  \item \textbf{Legal \& compliance.} CourtEdge is \textbf{analytics only --- it never accepts
    wagers}, which keeps it out of gambling-operator licensing. It must still respect
    state-by-state advertising rules, affiliate disclosure, responsible-gambling messaging,
    age-gating (21+), and each book's Terms of Service.
  \item \textbf{Data-acquisition cost \& risk.} \texttt{nba\_api} is unofficial and
    rate-limited; The Odds API free tier is small ($\sim$500 req/mo), so polling for line
    movement gets expensive at scale --- cost scales with \emph{polling frequency $\times$
    games $\times$ books}.
  \item \textbf{Cold-start.} Historical seasons are needed to backfill before the model is
    credible, and a CLV track record needs time to accumulate.
  \item \textbf{Competition \& moats.} Incumbents are well-funded and the underlying data is
    largely public --- the moat is weak. We are honest about this: the edge is focus,
    transparency, and price, not data exclusivity.
  \item \textbf{Unit economics.} At 100 users the binding cost is the odds-API tier + a single
    small VM (the demo runs on one workstation); at 1{,}000 users odds polling and DB load
    dominate, motivating caching, a tapering poll cadence, and a paid odds tier. Break-even is
    driven by odds-API spend, which is controllable by polling discipline.
\end{itemize}

\subsection{Data sources}
\begin{center}\small
\begin{tabular}{@{}lll@{}}
\toprule
\textbf{Source} & \textbf{What} & \textbf{Access / risk} \\
\midrule
\texttt{nba\_api} & game logs, play-by-play & free wrapper; unofficial, rate-limited \\
The Odds API & moneyline across US books & REST; free tier $\sim$500 req/mo \\
\bottomrule
\end{tabular}
\end{center}
\noindent We ingest five recent seasons (2020--21 \dots 2024--25): \textbf{5{,}995 games} and
\textbf{12{,}000 team-game rows}. For the big-data processing demo we ingest the
\textbf{full 2023--24 play-by-play: 598{,}705 event rows across 1{,}230 games}
($\sim$486 events/game).

\subsection{Storage --- why these tools fit the scale}
The raw zone is columnar \textbf{Parquet}, partitioned by season --- efficient for Spark reads
and the ``data lake'' story; the documented production path swaps it for MinIO/S3. The
\textbf{warehouse} holds curated, query-served tables; we use SQLite by default for
zero-friction reproducibility, and SQLAlchemy makes the switch to PostgreSQL a one-line
\texttt{DATABASE\_URL} change. Play-by-play stays in Parquet (too row-heavy for a transactional
store) and is processed by Spark.

\subsection{Processing --- the legitimate big-data core}
\textbf{PySpark} (running \texttt{local[*]}) builds rolling pre-game team-form features with
\emph{window functions} partitioned by team and ordered by date, using
\texttt{rowsBetween(-N,\,-1)} so every feature uses only the \emph{previous} $N$ games
(no leakage). The same DataFrame code aggregates the \textbf{598{,}705-row} play-by-play table
into per-game and per-team descriptors (lead changes, max margin, OT flag, FG\%, threes)
--- the genuinely distributed workload; identical code scales to a cluster and to all-time PBP
($\sim$10M+ rows). \textbf{Elo} ratings are computed sequentially in pandas (each game updates
ratings, with season-to-season regression toward the mean).

\paragraph{Model.} An \textbf{XGBoost} classifier over ten features (Elo expectation and
difference, rolling net-rating, points, and win-rate differentials, rest days, back-to-back
flags) with a
strictly time-ordered split: train on the oldest games, calibrate on a validation slice with
\textbf{isotonic regression}, and test on the most recent season. Calibration is essential
because edge $=$ model prob $-$ market-implied prob is only meaningful when the model
probability is well calibrated.

\subsection{The quant layer (exact formulas --- this is the product)}
American odds $o$ convert to \textbf{implied probability} and \textbf{decimal odds} as
\[
  \text{imp}(o)=\begin{cases}\dfrac{100}{o+100}, & o>0\\[6pt]\dfrac{-o}{-o+100}, & o<0\end{cases}
  \qquad
  d(o)=\begin{cases}\dfrac{o}{100}+1, & o>0\\[6pt]\dfrac{100}{-o}+1, & o<0.\end{cases}
\]
A two-way market is \textbf{de-vigged} to fair probabilities by
$p^{\text{fair}}_A=\text{imp}_A/(\text{imp}_A+\text{imp}_B)$. With model probability $p$ and
$b=d(o)-1$:
\[
  \underbrace{\text{edge}=p-\text{imp}(o)}_{\text{displayed}},\quad
  \underbrace{\text{EV}=p\,b-(1-p)}_{\text{per unit staked}},\quad
  \underbrace{f^{\star}=\frac{b\,p-(1-p)}{b}}_{\text{Kelly fraction}} .
\]
We bet only when EV $>0$, stake $0.25\times$ Kelly capped at $5\%$ of bankroll, and measure
\textbf{CLV} $=\text{imp}(\text{closing})-\text{imp}(\text{taken})$ (positive $\Rightarrow$ you
beat the close). All functions are pure and unit-tested in \texttt{src/model/quant.py}.

\subsection{Delivery}
A \textbf{Streamlit} dashboard serves three pages: \emph{Today's Board} (model probability vs.\
market, best price across books, edge, EV, Kelly stake), \emph{Model Performance} (out-of-sample
metrics + calibration curve --- the trust layer), and \emph{How It Works}. When an Odds API key
is set the board scores live upcoming games; off-season or keyless it falls back to a clearly
labelled \emph{sample board} built from real held-out predictions, so the UI always renders and
the edge math stays real. The app is deployed on a school workstation (see page 1).

\subsection{Honest scoping and the scaling path (10$\times$ / 100$\times$)}
What was built is a complete \emph{thin slice} end-to-end. Deliberately descoped items map
directly to the production architecture: SQLite $\rightarrow$ Postgres + MinIO/S3; Spark
\texttt{local[*]} $\rightarrow$ a cluster (YARN/K8s) over all-time play-by-play
($\sim$10M+ rows); Streamlit $\rightarrow$ FastAPI + Next.js with auth; and a daily Airflow DAG
plus a Kafka odds stream for live line-movement and automated CLV capture. Because the Spark
code is plain DataFrame/window-function code, it scales unchanged.

\paragraph{Reproducibility.} The repository commits the built warehouse
(\texttt{data/courtedge.db}) and model artifacts (\texttt{data/artifacts/}) so a fresh clone
serves in $\sim$1 minute via \texttt{requirements-serve.txt} (no Spark/Java). A full rebuild is
one command --- \texttt{bash run\_pipeline.sh} --- and individual steps are documented in the
README. API keys are read from \texttt{.env} (git-ignored) with a committed \texttt{.env.example}.

\section{Conclusion}
CourtEdge turns public NBA stats and sportsbook odds into a refined, monetizable data product
for a specific, paying segment, defended by transparent methodology rather than hype. The
technical core --- distributed feature engineering over $\sim$599k play-by-play events, a
calibrated win-probability model evaluated without leakage, and a unit-tested quant layer ---
is delivered through a deployed dashboard. We are honest about the system's limits: a highly
efficient market, a weak data moat, and a model that only modestly beats Elo. The value is not
a magic edge but better prices, correct sizing, and an auditable CLV record.
